  • 可建立半自动评估：MEMORY.md 记录评估清单

半自动评估 MEMORY.md 评估日志，人工触发填写

PHASE 2 迁移方案选型：纯脚本 vs 模板 + Skill
问题：如何将项目专属的 .claude/ 架构迁移到新项目？脚本硬编码还是 LLM 智能适配？

方案 A：纯脚本 claude-init.js

检测 → 正则匹配 → 字符串拼接 → 写文件
✅ 确定性输出，只需 Node.js
❌ 模板嵌在 JS 字符串里，维护体验差

方案 B：模板仓库 + Skill ✓

模板 .md → Skill 读取 → LLM 适配 → 写文件
✅ 模板所见即所得，直接编辑 .md
✅ LLM 理解上下文，边缘情况灵活处理
决策
纯脚本 作为验证原型实现，但不是最终方案
模板 + Skill 内容与逻辑分离，维护体验决定性优势
↓
PHASE 3 与 /init 命令的关系设计
问题：适配 Skill 和 Claude Code 内置的 /init 如何共存？
  • /init 只做一件事：分析项目 → 生成 CLAUDE.md
  • /init 不生成 rules / agents / skills
  • 适配 Skill 的能力是 /init 的超集
全新项目 → /adapting-project-structure → 生成完整 CLAUDE.md + 适配全部文件
已有 /init → 读取已有 CLAUDE.md → 合并而非覆盖
已有自定义 → 不覆盖 → 仅追加缺失段

持久化设计

docs/task-queue.md = 文件而非内存
✓ /loop 跨会话恢复
✓ compaction 不丢进度
✓ 多人会话共享队列

✓ 复用 /batch 的能力

  • 并行 agent spawn

  • 进度表跟踪

PHASE 6 模板内容设计："完整待裁剪"而非占位符
问题：模板文件用什么格式？占位符 {{PM}} 还是完整内容？

占位符方案 ❌

Install: {{pm_install}}
## {{framework}} Conventions
{{if_react}}...{{endif}}
需要正则替换，边缘情况难处理

完整待裁剪 ✓

## React → 保留
## Vue 3 → 删除
## Angular → 删除
LLM 理解"保留匹配的，删除其他的"
洞察

LLM 的语义理解力 > 正则匹配。让 LLM 做裁剪决策，模板只负责提供完整的内容覆盖。

PHASE 7 闭环执行设计
READ → IMPLEMENT → VERIFY → (FIX → VERIFY) × 3 → COMMIT
  • 每个任务必须独立可验证（build + test）
  • 最多 3 次 fix-and-retry，超过标记 [!] blocked
  • 持久化到 docs/task-queue.md，/loop 跨会话恢复
  • 任务状态：[x] completed → [-] in-progress → [!] blocked → [ ] pending
↓
方法论 提炼的 5 条设计原则
01
内容与逻辑分离
模板是纯 .md 内容，适配逻辑由 LLM 执行，不在内容中嵌入代码逻辑。
02
渐进式信息披露
rules 自动加载（核心约束），reference 按需加载（详细规范）。
03
持久化优于内存
任务队列是文件，不依赖会话上下文，天然支持跨会话和 /loop。
04
超集替代而非并存
当新能力完全覆盖旧能力时，吸收替代比并存更简洁。
05
复用机制而非调用命令
/batch 的 Plan Mode 会中断自主流程，但其底层机制可直接复用。
